# Design: TypeScript Extraction from Go Templates

## Problem

Go templates embed JavaScript inside `<script>` blocks, injecting Go values
via `{{.Field}}` expressions. These values are invisible to the TypeScript
compiler, so type errors in the JavaScript (wrong field names, wrong
assumptions about data shape) are only caught at runtime.

AI agents that generate frontend code for Go-template-driven apps hit this
especially hard: the agent writes JS that consumes template-injected data,
but there's no compiler feedback to catch mistakes before the code ships.

## Goal

Let `tsc` validate JavaScript inside Go templates by:

1. Extracting `<script>` blocks from `.gohtml` / `.tmpl` files
2. Replacing `{{.Field}}` expressions with typed TypeScript placeholders
3. Writing the result as `.ts` files that `tsc` can check

## Example

Given this Go code and template:

```go
type Page struct {
    Title string
    Items []Item
}

type Item struct {
    Name  string
    Price float64
}

_ = templates.ExecuteTemplate(w, "index.gohtml", Page{})
```

```html
<!-- index.gohtml -->
<h1>{{.Title}}</h1>
<script>
  const items = {{.Items}};
  items.forEach(item => {
    console.log(item.Naem); // typo — should be .Name
  });
</script>
```

`check-templates` would generate:

```typescript
// index.gohtml.ts — auto-generated by check-templates
// Source: index.gohtml <script> block (lines 3–8)

const items: Array<{ Name: string; Price: number }> = undefined!;
items.forEach(item => {
  console.log(item.Naem); // tsc ERROR: Property 'Naem' does not exist
});
```

Running `tsc --noEmit` catches the typo at build time.

## Type Mapping: Go to TypeScript

| Go type | TypeScript type |
|---------|-----------------|
| `string` | `string` |
| `int`, `int8`–`int64`, `uint`–`uint64`, `float32`, `float64` | `number` |
| `bool` | `boolean` |
| `[]T` | `Array<T>` |
| `[N]T` | `Array<T>` |
| `map[K]V` | `Record<K, V>` (K must be `string` or `number`) |
| `struct { ... }` | `{ Field: Type; ... }` |
| `*T` | `T \| null` |
| `interface{}` / `any` | `unknown` |
| `json.RawMessage`, `[]byte` | `unknown` |
| `time.Time` | `string` (JSON marshals to RFC 3339) |
| `encoding/json.Number` | `number \| string` |
| named type | resolved to underlying structure |

Struct fields follow `json` tag rules:

- `json:"name"` → uses `name` as the key
- `json:"-"` → omitted
- `json:",omitempty"` → field becomes optional (`Field?: Type`)
- Unexported fields without json tags → omitted (not marshalled)

## Template Expression Replacement

### Simple field access

```
{{.Title}}     →  (undefined! as string)
{{.Count}}     →  (undefined! as number)
{{.Items}}     →  (undefined! as Array<{ Name: string; Price: number }>)
```

The replacement is a typed `undefined!` cast. This satisfies tsc's type
checker without producing runtime code (these files are never executed).

### Chained field access

```
{{.User.Email}}  →  (undefined! as string)
```

The tool already resolves chained field types. The final type is used.

### Pipelines and functions

```
{{.Items | json}}   →  (undefined! as unknown)
{{printf "%s" .X}}  →  (undefined! as string)
{{len .Items}}      →  (undefined! as number)
```

For known functions (`len`, `printf`, etc.) the return type is known. For
custom FuncMap functions, the Go return type is resolved and mapped. If a
function's return type can't be determined, `unknown` is used.

### Assignment

```
{{$x := .Title}}
...later in <script>...
var name = {{$x}};
```

Variable assignments are tracked (the tool already does this) and the
type of `$x` is resolved to `string`.

## Control Flow Inside `<script>` Blocks

This is the hardest part. Template control flow changes what code exists
and what type dot (`.`) has.

### Phase 1: Ignore control flow

Strip template control-flow actions and emit only the "happy path":

```html
<script>
  {{if .ShowDebug}}
  console.log("debug:", {{.DebugInfo}});
  {{end}}
  const title = {{.Title}};
</script>
```

Generates:

```typescript
console.log("debug:", (undefined! as string));
const title: string = undefined!;
```

Both branches are emitted and type-checked. This may produce some false
positives (code that references a variable only defined in one branch)
but catches most real type errors.

### Phase 2: Model control flow (future)

```
{{if .X}}        →  if (true) {
{{else}}         →  } else {
{{end}}          →  }
{{range .Items}} →  for (const __dot of (undefined! as Array<ItemType>)) {
                     // dot is now ItemType inside this scope
{{end}}          →  }
{{with .User}}   →  { const __dot = undefined! as UserType;
{{end}}          →  }
```

This preserves scoping and gives `{{range}}` the correct element type
for dot. It's significantly more complex to implement because:

- Template `{{range}}` reassigns dot to the element type
- `{{with}}` reassigns dot to the pipe result (if non-zero)
- Nested scopes need TypeScript block scoping to match

This is worth doing eventually but not required for a useful v1.

## HTML Parsing

To find `<script>` blocks, we need to parse the static HTML fragments
between template actions. Approach:

1. Walk the template parse tree and collect `TextNode` values (the raw
   text between `{{ }}` actions)
2. Concatenate them with placeholder markers for each action
3. Feed the result through Go's `golang.org/x/net/html` tokenizer to
   find `<script>` and `</script>` boundaries
4. Extract the content between those boundaries, re-expanding the
   placeholder markers back to typed TypeScript expressions

Edge cases:
- Template actions inside HTML attributes: `<script src="{{.URL}}">`
  — these are not script *content*, skip them
- Multiple `<script>` blocks per template — generate one TS block per
  script block, or concatenate with comments
- `<script type="application/json" id="...">` — the content is JSON, not
  JS; these blocks are handled via branded types (see below)

## Branded Types for `<script type="application/json">` Blocks

Go templates often serialize data into `<script type="application/json">` blocks
that are later consumed by JavaScript via `getElementById` and `JSON.parse`.
Without type information, the parsed result is `any`.

The ts-extract output uses **branded types** to thread the Go type through
the DOM API chain so TypeScript knows the parsed result without casts.

### Generated infrastructure

When JSON blocks are present, the output emits:

```typescript
declare const __brand: unique symbol;
type JSONString<T> = string & { [__brand]: T };
interface JSON { parse<T>(text: JSONString<T>): T; parse(text: string): any; }
```

`JSONString<T>` is a **branded type** — structurally a `string`, but the
phantom `[__brand]` property carries the parsed type `T`, making it
nominally distinct from plain strings.

### Per-block `getElementById` overloads

For each `<script type="application/json" id="...">` block, an overload
is emitted on `Document`:

```typescript
interface Document {
  getElementById(id: "page-data"): (HTMLScriptElement & { textContent: JSONString<PageData> }) | null;
}
```

### End-to-end type flow

Given a template block like:

```html
<script type="application/json" id="page-data">{{.PageData}}</script>
```

Where `PageData` is a Go struct, the type flow is:

1. `document.getElementById("page-data")` → `HTMLScriptElement & { textContent: JSONString<PageData> }`
2. `el.textContent` → `JSONString<PageData>` (branded string carrying the type)
3. `JSON.parse(el.textContent)` → `PageData` (the branded `JSON.parse<T>` overload matches)

TypeScript infers the correct type at each step — no manual type
assertions needed. A typo like `data.Naem` triggers a tsc error.

## Branded Types for `data-*` Attributes

Elements with `id` and `data-*` attributes containing template actions get
typed `getElementById` overloads with narrowed `dataset` properties.

### Example

```html
<div id="app" data-config='{{.Config | json}}' data-name="{{.Name}}">
```

Generates:

```typescript
interface Document {
  getElementById(id: "app"): (HTMLElement & { dataset: { config: JSONString<Config>; name: string } }) | null;
}
```

JSON-piped data attributes use `JSONString<T>` so `JSON.parse(el.dataset.config)`
returns the typed result. Scalar attributes are typed as `string` (since
HTML data attributes are always strings in the DOM).

The `data-*` attribute name is converted to camelCase for `dataset` access
following the standard DOM convention (`data-user-id` → `dataset.userId`).

## Branded Types for `<meta>` Tags

`<meta>` tags with `name` and `content` attributes containing template actions
get typed `querySelector` overloads.

### Example

```html
<meta name="csrf-token" content="{{.CSRFToken}}">
<meta name="page-config" content='{{.Config | json}}'>
```

Generates:

```typescript
interface Document {
  querySelector(selector: 'meta[name="csrf-token"]'): (HTMLMetaElement & { content: string }) | null;
  querySelector(selector: 'meta[name="page-config"]'): (HTMLMetaElement & { content: JSONString<Config> }) | null;
}
```

For JSON-piped content, `JSON.parse(el.content)` returns the typed result.
For scalar content, `.content` is typed as `string`.

## Branded Types for `<input type="hidden">`

Hidden form inputs with `id` and `value` attributes containing template
actions get typed `getElementById` overloads.

### Example

```html
<input type="hidden" id="user-data" value='{{.User | json}}'>
```

Generates:

```typescript
interface Document {
  getElementById(id: "user-data"): (HTMLInputElement & { value: JSONString<UserType> }) | null;
}
```

The type flow: `getElementById("user-data")` → `.value` → `JSONString<UserType>`
→ `JSON.parse(el.value)` → `UserType`.

## CLI Interface

```sh
# Generate .ts files alongside templates
check-templates -o ts-extract ./...

# Output directory (default: next to template files)
check-templates -o ts-extract -ts-outdir ./generated ./...

# Generate .d.ts declaration files only (no script extraction)
check-templates -o dts ./...
```

### Output modes

**`ts-extract`** — full script block extraction:
- Reads each template's `<script>` blocks
- Replaces `{{.X}}` with typed placeholders
- Writes `<template>.ts` files

**`dts`** — simpler, declaration-only mode:
- Emits one `.d.ts` file mapping template names to their data types
- Useful even without script extraction: agents and developers can
  import the types when writing separate `.ts` files

```typescript
// templates.d.ts — auto-generated by check-templates
declare namespace TemplateData {
  interface IndexGohtml {
    Title: string;
    Items: Array<{
      Name: string;
      Price: number;
    }>;
  }
  interface HeaderGohtml {
    NavItems: Array<{ Label: string; Href: string }>;
    LoggedIn: boolean;
  }
}
```

## Integration

### CI pipeline

```yaml
steps:
  - run: go build ./...
  - run: check-templates ./...              # type-check templates
  - run: check-templates -o ts-extract ./... # extract TS files
  - run: npx tsc --noEmit                   # validate JS in templates
```

### tsconfig.json

Generated `.ts` files need to be in scope. Either place them in an
included directory or add an explicit include:

```json
{
  "include": ["src/**/*.ts", "templates/**/*.gohtml.ts"]
}
```

### AI agent workflow

1. Agent receives a task involving frontend changes
2. Agent writes/modifies Go handler + template
3. `check-templates ./...` catches template type errors
4. `check-templates -o ts-extract ./...` generates typed TS
5. `tsc --noEmit` catches JavaScript type errors
6. Agent gets compiler feedback and self-corrects

The agent never needs to know about the Go-to-TS bridge — it just sees
compiler errors and fixes them, the same way it would with any TS project.

## Implementation Order

| Phase | What | Effort |
|-------|------|--------|
| 1 | `-o dts` — emit `.d.ts` from resolved Go types | Medium |
| 2 | HTML `<script>` block detection via `x/net/html` tokenizer | Medium |
| 3 | `-o ts-extract` — replace `{{.X}}` with typed placeholders, strip control flow | High |
| 4 | Control flow translation (`{{if}}` → `if`, `{{range}}` → `for`) | High |

Phase 1 is independently useful and doesn't require HTML parsing. Phase 2
and 3 depend on each other. Phase 4 is an enhancement.

## Open Questions

1. **Should generated files be `.ts` or `.js` with JSDoc annotations?**
   `.ts` is more natural for tsc but requires a TypeScript setup. JSDoc
   annotations work in plain `.js` files with `checkJs: true`.

2. **How to handle `template.HTML` / `template.JS` types?** These are
   `string` typedefs in Go that bypass escaping. In the TS output they'd
   just be `string`, but we could add a branded type to flag them.

3. **Inline `<script>` vs external `.js` files?** This design focuses on
   inline scripts but also covers data channels consumed by external JS:
   `data-*` attributes, `<meta>` tags, and `<input type="hidden">` all
   get branded type overloads. For other external-JS patterns, the `dts`
   mode may be more appropriate.

4. **Source maps?** ~~Generated `.ts` files could include source map
   comments pointing back to the `.gohtml` line numbers.~~ **Implemented.**
   Each `ScriptBlock` now carries a `LineMap` that maps TS output lines
   back to template source lines. See the Source Mapping section below.

## Source Mapping

Each `ScriptBlock` includes a `LineMap []int` where `LineMap[i]` is the
1-based template source line corresponding to TS output line `i+1`.

### How it works

During the concatenation phase (where template text nodes and action
placeholders are assembled into a single HTML string), a parallel
`sourcePos []int` array is built that records the template byte offset
for every byte in the concatenated string. Each `TextNode` contributes
its `Pos` (byte offset in the template source) and each `ActionNode`
placeholder maps to its action's `Pos`.

When a `<script>` block region is extracted, the `sourcePos` array is
used to determine the template source line for each line in the script
content. This produces the `LineMap`.

### Column mapping

In addition to line mapping, each `ScriptBlock` carries per-line column
adjustment data (`ColMap` and `ColOffset`) that tracks how placeholder
replacements shifted columns within each line.

Each replacement is recorded as a `colSpan`:

```go
type colSpan struct {
    TSCol   int // 1-based column in TS output where replacement starts
    OrigCol int // 1-based column in template source where {{...}} starts
    OrigLen int // length of original {{...}} action
    NewLen  int // length of replacement (e.g. "(undefined! as string)")
}
```

The `ColOffset` field records how many leading characters were trimmed
from the first line (due to `TrimSpace`), so that non-replacement columns
on the first line can be correctly offset back to template coordinates.

For any TS column position, `adjustCol` walks the spans to determine:
1. **Before a replacement**: apply cumulative shift + column offset
2. **Inside a replacement**: clamp to the original `{{...}}` start column
3. **After all replacements**: apply total shift + column offset

### Usage

The `ScriptBlock.MapLocation(tsLine, tsCol)` method translates a
TypeScript diagnostic location (1-based line and column, as reported by
`tsc`) to a `SourceLoc` with the corresponding template line and column:

```go
// tsc reports: test.gohtml.ts(3,15): error TS2339: Property 'Naem' ...
block := result.ScriptBlocks[0]
loc := block.MapLocation(3, 15)
// loc.Line = template source line, loc.Col = adjusted template column
fmt.Printf("%s:%d:%d: %s\n", block.TemplateFile, loc.Line, loc.Col, tscMessage)
```

### Limitations

- Columns that fall inside a replacement (e.g. within `(undefined! as Type)`)
  are clamped to the start of the original `{{...}}` action. This is the
  best we can do since the replacement text has no character-level
  correspondence to the original action content.
- Control flow flattening emits all branches sequentially. Lines from
  flattened `{{if}}`/`{{range}}`/`{{with}}` blocks map correctly to
  their original template positions.
